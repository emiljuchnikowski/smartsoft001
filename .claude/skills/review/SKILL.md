# Review Skill

Run multi-dimensional code review covering quality, tests, security, and performance.

## Usage

```
/review [linearTaskId]
/review --staged
/review --diff=main
```

## Execution Checklist

- [ ] **1. Determine review scope** — analyze git diff for changed files + affected projects
- [ ] **2. Quality review** — lint, type-check, code style, import ordering
- [ ] **3. Test review** — coverage, test quality, AAA pattern compliance
- [ ] **4. Security review** — OWASP top 10, dependency audit, secrets detection
- [ ] **5. Performance review** — bundle impact, change detection, memory leaks
- [ ] **6. Synthesize findings** — merge all findings, deduplicate, prioritize
- [ ] **7. Post to Linear** — create review comment in Polish (if linearTaskId provided)
- [ ] **8. Present to user** — show findings with severity ratings

## Instructions

### Step 1: Determine Review Scope

**If `linearTaskId` provided:**

1. Find related commits via `git log --all --grep="<taskId>"`
2. Determine changed files via `git diff main...<branch>`

**If `--staged`:**

1. Use `git diff --cached --name-only`

**If `--diff=<branch>`:**

1. Use `git diff <branch>...HEAD --name-only`

### Step 2: Quality Review

Delegate to `shared-style-enforcer` agent:

- Run linting: `nx lint <project>` for affected projects
- Run type-check: `npx tsc --noEmit`
- Check import ordering, naming conventions
- Verify Angular modern APIs (signals, @if/@for, inject())

### Step 3: Test Review

Delegate to `shared-test-runner` and `shared-coverage-enforcer`:

- Run tests: `nx test <project> --coverage`
- Check coverage meets 80% minimum
- Verify AAA pattern compliance
- Check test naming: `describe('@smartsoft001/...: ClassName')`

### Step 4: Security Review

Delegate to `shared-security-scanner`:

- Check OWASP Top 10 in changed code
- Run `npm audit`
- Check for hardcoded secrets
- Verify input validation

### Step 5: Performance Review

- Check new imports and bundle impact
- Verify `track` in all `@for` loops
- Check for memory leaks (unsubscribed observables)
- Verify OnPush change detection usage

### Step 6: Synthesize and Report

```markdown
## Code Review Report

### Overall Verdict

[PASS / PASS_WITH_WARNINGS / BLOCK]

### Summary

| Dimension   | Critical | Medium | Low | Status     |
| ----------- | -------- | ------ | --- | ---------- |
| Quality     | X        | Y      | Z   | PASS/BLOCK |
| Tests       | X        | Y      | Z   | PASS/BLOCK |
| Security    | X        | Y      | Z   | PASS/BLOCK |
| Performance | X        | Y      | Z   | PASS/BLOCK |

### Critical Issues

- [issue] — file: path/to/file.ts:line

### Warnings

- [warning] — file: path/to/file.ts:line

### Passed Checks

- [x] Check passed

---

_Review generated by Claude Code_
```

## Guidelines

1. **All Linear comments in Polish**
2. **Every finding must have severity rating** — CRITICAL/HIGH/MEDIUM/LOW
3. **Recommendations must be actionable** — suggest fixes, not just flag issues
4. **Respect existing patterns** — don't flag established conventions
5. **Be constructive** — balance criticism with positive observations
